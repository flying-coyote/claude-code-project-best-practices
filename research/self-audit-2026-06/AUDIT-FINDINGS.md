# Self-audit findings — claude-code-project-best-practices

This is the consolidated read of the six-lens self-audit run against the best-practices repo on 2026-06-21. Each lens applied a set of Miessler "Fable" prompts to the repo itself rather than to a target project, and the striking thing across all six is the convergence: a single broken mechanism is the top recommendation of five of the six lenses, and a single conceptual gap (the audit measures what a project *has*, never *why*) sits underneath nearly every "the repo doesn't apply its own advice to itself" finding. The repo's content is genuinely current and its design is sound — the wrongness lives in the machinery, the meta-docs, and one missing dimension of the audit itself, not in the analysis claims.

The findings below are ordered by leverage, not by lens. The per-lens summaries follow, then the structural gap that the refactor phase should treat as the centerpiece.

## Top recommendations

1. **Repoint the measurement-expiry gate from `patterns/` to `analysis/` and make it blocking.** `scripts/check-measurement-expiry.py:196` defaults `--patterns-dir patterns`, and `scripts/generate-tools-tracker.py:30` hard-codes `PATTERNS_DIR = Path("patterns")`, but `patterns/` was archived to `archive/patterns-v1/` in the March 2026 v2.0 reposition and the live measurement-claims now live in `analysis/`. The daily workflow `.github/workflows/tools-evolution-tracker.yml:57-60` runs the script with no `--patterns-dir` override, so CI scans 0 files, exits 0, and reports "all claims current" under a green badge — while pointing it at `analysis/` surfaces 2 claims in `analysis/mcp-patterns.md` that are 93 days overdue (revalidate 2026-03-20). The step also carries `continue-on-error: true` (line 61), so even a real failure would not block. Fix: change both defaults to `analysis` (or pass `--patterns-dir analysis`), drop `continue-on-error` for the expiry step, extend the checker to also read the top-level `revalidate-by` field (it currently parses only the nested `measurement-claims[].revalidate`), and triage the 2 overdue mcp-patterns claims. This is the single fix flagged by lenses 2, 3, 5, and 6 as their top recommendation, and by lens 1 as the cheapest mechanical route to catching the staleness and license drift. *Refactor target: loop-eng-elevation docs (evidence-based-revalidation operationalization), plus the workflow and two scripts.*

2. **Write the signal→doc sync linter that AUDIT-CONTEXT.md already says "must" exist, and wire it into CI.** `AUDIT-CONTEXT.md:10,242` asserts that routing rows and per-doc `applies-to-signals` frontmatter must stay in sync and that "a sync linter must whitelist these classes" — but no such linter is committed (grep for `applies-to-signals` across `scripts/`, `automation/`, `.github/` returns nothing; `automation/` holds only `generate_index.py`). The invariant is maintained by hand across 42 docs and ~78 distinct signal tokens, with a six-file manual integration checklist per new doc (`CONTRIBUTING.md:68-80`). This is the crown-jewel-protection fix: the adaptive routing audit is the repo's only genuinely unique asset, and its correctness currently rests on the maintainer remembering the checklist. A ~50-line deterministic linter that diffs signal tokens against routing keys with the documented whitelist, failing CI on any unreachable doc or dangling key, moves audit correctness from memory to a merge gate and is the highest-leverage bus-factor reduction available. *Refactor target: AUDIT-CONTEXT.md (and a new CI linter that enforces it).*

3. **Harden `claude-code.yml` Job 2 so an external GitHub user cannot reach a write-scoped agent.** `.github/workflows/claude-code.yml` Job 2 (`process-source-update`, lines 32-104) runs `claude-code-action` with `contents: write` + `pull-requests: write`, triggers on `issues: opened` gated only by a `source-update` label, fetches an attacker-controlled URL from the issue body, and passes only `--max-turns 30` with no `--allowedTools` restriction — the canonical injection-to-action chain the repo's own `safety-and-sandboxing.md:308` warns about. Fix (all config-only and reversible): gate the trigger on `github.event.issue.author_association` (OWNER/MEMBER), add an `--allowedTools` allowlist instead of unrestricted Bash, and require review on `master` via branch protection. This is the only finding where an outside party can reach a write-scoped agent through attacker-controlled text. *Refactor target: security (the deployed workflow surface), informed by safety-and-sandboxing.md.*

4. **Add an injection boundary to the audit prompt itself.** The shipped audit (`AUDIT-CONTEXT.md:20-68`) has the auditing agent `cat`/`grep`/`find` over an arbitrary target project's files with no instruction to treat that content as untrusted data — a grep for injection/untrusted/treat-as-data language across README, ONE-LINE-PROMPT, and AUDIT-CONTEXT returns nothing. This bites hardest on Opus 4.8, whose injection robustness regressed versus 4.7, a fact this very repo documents (`safety-and-sandboxing.md:246-269`). Fix: one line in the ONE-LINE-PROMPT.md prompt block and the AUDIT-CONTEXT.md preamble instructing the agent to treat all target files and fetched pages as DATA, never instructions, and to report any redirection attempt as a finding. Pure text addition, one-line revert. *Refactor targets: ONE-LINE-PROMPT.md and AUDIT-CONTEXT.md.*

5. **Run the audit end-to-end against this repo and one external repo, and commit both outputs under `examples/`.** The core value proposition — one prompt, cited recommendations — has no demonstrated run anywhere in the repo; searching for the audit's own output markers (`audit-date:` / `repo-name:`) finds only the template in ONE-LINE-PROMPT.md, and `examples/` holds only an archived v1 research-project README. A committed self-audit is a true dogfood that would have caught the staleness and license findings mechanically, and a second external run gives contributors a regression fixture. (This synthesis is itself a partial discharge of the self-audit half — but the *audit-format* run, producing the structured output the product promises, is still owed.) *Refactor target: intent-alignment-audit.md / examples (demonstrate the mechanism), feeding the regression-fixture work below.*

6. **Add deterministic routing-fixture regression tests.** No fixture pins "project with signals X → triggers keys Y → fetches docs Z" anywhere outside `node_modules`, even though the repo documents regression testing and golden-answer comparison (`analysis/agent-evaluation.md:59-64,100-106`) and states "you can't improve what you can't measure" (`:22`). `PLAN.md:38` deferred CI regression tests "until Anthropic publishes guidance," but that reason applies only to the LLM-judged "are the recommendations good" half — the deterministic routing-invariant half (3-5 synthetic repos with expected signal-key + fetched-doc manifests, checked model-free) needs no vendor guidance and shares a harness with the sync linter in recommendation 2. *Refactor target: loop-eng-elevation docs (the eval/regression discipline), plus a new tests/ fixture harness.*

7. **Reconcile the repo's self-model: one canonical "what this is," a current ARCHITECTURE.md, and a single licence.** Three drift clusters travel together here. (a) The repo gives three incompatible answers to "what is this" — README's portable audit, ARCHITECTURE.md's "Consumer Reports analytical layer," package.json's "curated collection of patterns," plus DECISIONS.md 1-8 describing the superseded pre-v2.0 bootstrap product without a superseded banner. (b) ARCHITECTURE.md is the one genuinely stale doc, frozen at the April "26 analysis documents" snapshot while README/PLAN/INDEX/DECISIONS all say 42-43, with wrong sibling-repo URLs (`anthropics-solutions/...`, `obraun-cl/...` vs the real `affaan-m/...`, `obra/...`) and a stale 110K star count vs README's 119K. (c) Licence is asserted three ways (README + CONTRIBUTING = MIT, package.json = ISC) with no LICENSE file, and package.json carries version 1.0.0, a phantom `main: index.js`, and an empty author. Fix: make README's audit framing canonical and have the weaker surfaces defer to it; rewrite ARCHITECTURE.md to the 42-doc reality or demote it to a thin pointer at the auto-generated INDEX.md; add a root MIT LICENSE file and set package.json to MIT with a corrected version/author. The ECC star-count drift across five files (110K vs 119K) is the same decaying-snapshot smell and should follow Decision 9 — describe ECC's role qualitatively, keep one dated count in SOURCES.md only. *Refactor target: README/ARCHITECTURE.*

8. **Stamp or retire the orphaned artifacts the repo otherwise tracks well.** `V2-COMPLIANCE-MATRIX.md` (root, dated 2026-03-31) is a finished v2.0-migration snapshot still linked from INDEX.md as if current, scoring repos like `security-architect-mcp` (abandoned May 2026) and `blog` (retired 2026-05-24) as GOOD — and a static cross-repo matrix is exactly the hand-built artifact the on-demand audit was designed to replace. The two `graphify_*` scripts are wired into no workflow but carry no "reference pattern, not run in CI" marker. The `tools-evolution-tracker.yml` workflow auto-commits a tracker generated from an empty scan while holding `contents: write` — either fix the generator (recommendation 1) or demote it to `issues: write` until verified. And `.github/workflows/README.md` documents 3 of 5 workflows, omitting the two that run the Python scripts and wrongly claiming "Secrets Required: None" while `anthropic-blog-rss.yml` uses `ANTHROPIC_API_KEY`. Also fix the stale `patterns/` docstrings in the three scripts, the broken `analyze-blog-post.py` model ID (`claude-opus-4.6` → a valid current ID), the `.mcp.json` entry pointing at the archived MCP server, and the missing `DEPRECATIONS.md`/`DOGFOODING-GAPS.md` the scripts reference. *Refactor targets: loop-eng-elevation docs and the workflow/script housekeeping.*

## Per-lens summaries

### Lens 1 — Intent and goal (Fables #1, #3, #9, #13)
The repo knows its job — the portable adaptive audit — and its load-bearing files (README + AUDIT-CONTEXT.md + ONE-LINE-PROMPT.md) are built for it; the routing design is the positive baseline to preserve. The weakness is self-model accuracy: the goal is stated three incompatible ways across the identity docs, ARCHITECTURE.md is a stale self-contradicting snapshot that fails the staleness check the repo teaches, and the licence is declared three ways with no LICENSE file. The named "one real constraint" is that the audit has never been demonstrated end-to-end — no captured run, not even a dogfood — so the value proposition rests on an untested 65-line protocol. Top rec: run and commit a self-audit plus one external run, which also mechanically catches the staleness and licence drift.

### Lens 2 — Bitter lesson and "is the repo getting better" (Fables #2, #5)
Bitter-lesson self-awareness is genuinely strong — the repo tracks first-party feature absorption, runs an active retirement lane that retired `session-quality-tools.md` when `/insights` shipped, and keeps the prompt deliberately thin. But it cannot tell whether it is getting better: its one automated regression mechanism (the measurement-expiry gate) is dead-pointed at the deleted `patterns/` dir and reports all-clear while two claims are 93 days overdue; there is no routing regression fixture despite the repo documenting golden-answer comparison; and the promised sync linter does not exist. The minimal eval is three deterministic, model-free checks — fixed expiry gate, routing fixtures, sync linter — none of which need the Anthropic guidance PLAN.md:38 waits on. The bitter-lesson `harness-comprehensive` diagnostic is also applied only outward, never to the repo's own 42-doc footprint.

### Lens 3 — Memory, autonomy, policy (Fables #4, #6, #12)
The compounding-knowledge and decisions-into-policy machinery is real: DECISIONS.md is a true append-only log, CONTRIBUTING.md encodes the EMERGING→RETIRED lifecycle as policy, and `scheduled-and-looping-primitives.md` analyzes the autonomy ladder with measured detection predicates. The failure is intent-mechanism drift on the script side: the v2.0 reposition moved content to `analysis/` but left the enforcement scripts globbing `patterns/` and narrating the abandoned layout in their docstrings. The one workflow on the autonomous (`contents: write`) rung auto-commits a tracker built from an empty scan; the correct rung (RSS/source-monitoring as `issues: write` / escalate-to-human) shows the repo knows the pattern but didn't apply it to its own broken generator. PLAN.md's cadence table overstates live automation, and script escalation copy points at a `DEPRECATIONS.md`/`DOGFOODING-GAPS.md` that were never created. The clean policy-promotion candidate is the budget-ceiling-as-blast-radius judgment, currently sitting as a Tier-D gap rather than an emitted recommendation.

### Lens 4 — Security (Fables #7, #8)
The repo's published security content is accurate and current (OWASP MCP Top 10, the 4-layer permission stack, the Opus 4.8 injection regression with correct figures) — the gap is that it does not apply its own advice to its own deployed surface. The high-severity items: `claude-code.yml` Job 2 lets any GitHub user reach a `contents: write` + `pull-requests: write` agent via a labeled issue with no tool allowlist, and the audit prompt ingests target-project files with no injection boundary. Medium items: the copy-paste prompt asks users to blind-fetch a mutable `master` URL and run every shell command it contains (including unpinned `npx -y claude-doctor`); `.mcp.json` references an MCP server whose source is no longer tracked (stale-config, not live RCE — the archived server is read-only); and the RSS analyzer pipes untrusted web HTML into a Claude prompt unframed with a malformed/stale model ID. Genuinely clean: secrets hygiene, least-privilege on most workflows, the well-behaved RSS fetcher, and the read-only archived MCP server.

### Lens 5 — Where most wrong / what 10xs / bus-factor (Fables #10, #11, #14)
The content is current — every `revalidate-by` is in the future and the routed docs are fresh — so the most-wrong thing is not a claim but a mechanism: the daily cron enforcing the revalidation discipline has been silently no-op-ing against live content for ~3 months. ARCHITECTURE.md is the one stale doc (two versions behind at "26 documents"), the workflows README documents 3 of 5 workflows, and the licence drifts three ways with no LICENSE file. The 10x lever is the adaptive routing audit, whose currency rests on the documented-but-unimplemented signal→doc sync linter — committing that linter is the highest-leverage single change because it moves correctness from "the maintainer remembers" to "CI blocks the merge." Dead weight is minimal and mostly handled; only the two `graphify_*` scripts and the finished `V2-COMPLIANCE-MATRIX.md` lack status markers. Bus-factor reads moderate, well-mitigated on content, under-mitigated on mechanism (git history was not inspectable under the no-git guardrail, so the contributor-count read is UNVERIFIED).

### Lens 6 — Dev-loop and public surface (Fables #15, #16)
The write-loop has real, working gates (the local PostToolUse INDEX regen and auto-format, `stop-doc-check.sh`, `credential-scan.sh`, lint + link-check in CI) and the public surface is genuinely strong and consolidated — README/ONE-LINE-PROMPT/AUDIT-CONTEXT split cleanly by concern with honest count discipline (42 routable + 1 template) and no prompt drift. The biggest gap is an aspirational loop that rotted: the two expiry/tracker scripts still scan the archived `patterns/` dir, so the freshness gate runs green while verifying the empty set. Secondary: `generate-tools-tracker.py` is an orphaned generator writing to a path that only exists under `archive/`; the workflows README is stale and undercounts workflows and secrets; and INDEX regeneration works locally but is not an npm script or a CI gate, so an external fork that doesn't run the `.claude/` hook can land a doc without INDEX updating. package.json metadata is internally inconsistent (version, licence, phantom `main`).

## What the repo is missing structurally

Underneath the convergent mechanical findings sits one conceptual gap that the refactor phase should treat as the centerpiece: the audit inspects what a project **has**, never **why**. Every Signal Collection command in AUDIT-CONTEXT.md is a presence/absence or count check — does a `CLAUDE.md` exist, how many `.md` files, is `.mcp.json` present, is cron disabled — and every routing row maps the *existence* of a signal to a set of docs to fetch. That makes the audit excellent at telling a project what conventions it is missing and weak at telling it whether the conventions it has are the right ones for its intent. The same blind spot is exactly why the repo fails its own checks on itself: the expiry script scans a directory the repo no longer *intends* to use but still mechanically points at; ARCHITECTURE.md states a doc count the repo's actual structure has outgrown; the autonomy workflow holds a write rung nobody decided it should hold. A presence-based audit cannot catch intent-mechanism drift, because drift is precisely the case where what a project *has* (a `patterns/` glob, a "26 documents" claim, a `contents: write` permission) has come apart from *why it has it*. The refactor's highest-value addition is therefore not another signal but a new layer — an intent-alignment pass (a RETHINK/"why" dimension) that, for each load-bearing mechanism, asks what it is *for* and checks the mechanism against that stated intent, rather than only cataloguing its presence. That layer is what would let the audit catch on other projects the exact class of failure this self-audit caught on the repo itself, and it is the structural through-line connecting the broken expiry gate, the stale self-model, the unstated autonomy rung, and the missing injection boundary into a single root cause.
