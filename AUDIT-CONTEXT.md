# Audit Context: Signal → Advisory Routing Map

**Purpose**: This file is fetched by the [adaptive routing audit prompt](ONE-LINE-PROMPT.md) to route other projects to the analysis docs that apply to what they actually have, not all 27 at once. Each row states a verifiable signal, the docs to fetch, and the reason for the fetch.

**Design principles**:

- Routing is **inclusive** (multiple signals can trigger overlapping fetches) but **deduplicated** (fetch each doc only once even if multiple signals match).
- Signals are phrased as **observable facts** an agent can verify with a concrete command — not subjective judgments.
- **Always Fetch** rows fetch unconditionally, regardless of observed signals. They are non-negotiable baselines.
- Every analysis doc has a machine-readable `applies-to-signals` frontmatter field using the **signal vocabulary** in the rightmost column below. Routing rows and doc frontmatter must stay in sync.

**Evidence-tier reminder**: Tier A = Anthropic / primary observation. Tier B = expert practitioner / production validation. Tier C = community. See [analysis/evidence-tiers.md](analysis/evidence-tiers.md). Authority-weighted source list: [SOURCES-QUICK-REFERENCE.md](SOURCES-QUICK-REFERENCE.md).

### Edge Cases — handle silently, do not fail the audit

- **No `.claude/` directory**: signal = `harness-minimal` (if CLAUDE.md exists at repo root) or `claude-md-missing` (if not).
- **No git history or bare repo**: skip commit-pattern rows; note in output.
- **`settings.json` has no model field**: signal = `model-version-unknown`.
- **`claude doctor` unavailable (older CLI)**: skip session-diagnostic rows; note in output.
- **WSL or non-POSIX paths**: commands still work; treat any command failure as "signal not observed."
- **Any command times out or errors**: treat as signal not observed; do not fail the audit.
- **Markdown count returns 0**: no `md-corpus-*` signal triggers; the project is not a knowledge corpus.
- **Karpathy layout partially present** (e.g., `index.md` exists but no `raw/`): `vault-karpathy` does NOT match; the Lum1104 `/understand-knowledge` skill requires the full triple. Fall back to `/understand-anything:understand` recommendation in archetype-A.
- **`CLAUDE_CODE_DISABLE_CRON=1` observed**: signal = `cron-disabled`. Suppress `harness-loop-config`, `harness-scheduled-agent`, and `harness-goal-completion-loop` — the **local** scheduler is off, so loop-hardening advice would be noise. This does **not** bound cloud Routines (server-side, configured via `/schedule`): a local env var doesn't disable them, so still emit the cloud-Routine operator question.
- **Host-level signals** (`~/.claude/scheduled-tasks/`, `/goal` in `~/.claude/projects/` transcripts): user-home, not repo-local. Treat absence as "not observed," like session diagnostics — never fail the audit on them.
- **Cloud Routines leave no on-disk footprint**: absence of the unattended-execution signals does NOT prove the project runs nothing unattended. A pure-cloud Routine on Anthropic infrastructure is invisible to a repo-local audit — note this rather than asserting full coverage.
- **CLI older than v2.1.72** (from `claude --version`): scheduled tasks and `/loop` are unavailable; skip the unattended-execution rows.
- **Intent greps find nothing**: the intent signals are *evidence-of-absence* checks, and absence of evidence is not proof — a project's "why" may live in a wiki, an issue, a README paragraph the grep missed, or a maintainer's head. Treat `intent-undocumented` and `goal-drift-unmeasured` as **interview prompts, not verdicts**: when the grep comes up empty, do not assert the project has no intent; instead pose the Intent Interview below and let the human answer settle it. Phrase any finding "verify" (like the cloud-Routine and `generated-docs-no-drift-gate` blind spots), and never fail the audit on an intent signal.

## Intent Interview (when "why" can't be grepped)

The Signal Collection above measures what a project **has**; it cannot read **why** the project has it, which is the one thing a presence/absence audit is structurally blind to (see `analysis/intent-alignment-audit.md`). When the intent greps come up empty — no stated-why doc, no standing goal re-check — do **not** infer the project is aimless. A grep can't see intent that lives in a maintainer's head or a paragraph it didn't match. Instead, pose this short interview to a human and route on the answers. Keep it to the questions below (drawn from the four intent-and-goal prompts behind `intent-alignment-audit.md`); ask only the ones the greps left unresolved, and stop once the answers are concrete.

1. **What is this project for, in one sentence — and where is that written down?** (Fable #1, goal-orientation.) If the human gives a clear sentence but can't point to a doc that holds it, that's `intent-undocumented`: the "why" exists only in their head and won't survive a handoff. If three people on the project would answer differently, that's the self-model fragmentation the intent doc treats as the same gap.
2. **What would tell you this work has drifted off that goal — and what re-checks it on a cadence?** (Fable #13, the one real constraint.) If there's no standing instrument (no review, no skill, no scheduled "is this still serving the goal" pass), that's `goal-drift-unmeasured` — the project can only notice drift by accident.
3. **For each looping or scheduled mechanism you run, what is it *for*, and what decides it's still worth running?** (Fable #3, self-model accuracy.) A loop that no one re-evaluates is the strong-Act/stale-Orient pathology below — work keeps executing against a goal nobody re-checked. If the answer is "it just runs," that's `loop-without-rethink`.
4. **When the goal last changed, what mechanisms got left pointing at the old one?** (Fable #3, intent-mechanism drift.) This surfaces the exact failure the self-audit caught on this repo — a glob, a permission, a count that the project still *has* but no longer has a *reason* to have.
5. **If your typed notes carry a `type:`, who owns the list of allowed types, and what stops a new one from being invented ad hoc?** (Memory hygiene; only ask if `type:` frontmatter was observed without a registry.) No owning registry + no guard is `typed-memory-no-registry`. project1 is the worked positive case here — `01-knowledge-base/_type-registry.md` is the canonical list and `automation/lib/okf.py` is the pre-commit guard that blocks a commit on an off-registry type — so the interview answer to aim the project at is "a single parsed registry plus a commit-time guard," not a fourth ad-hoc convention.

Record the answers in the audit output and treat them as Tier-B evidence (human report of intent), citing the question number. The interview **resolves** an intent signal that grep left as "verify"; it does not by itself trigger a different fetch — the routing rows below still point every intent signal at `analysis/intent-alignment-audit.md`.

### Strong-Act / Stale-Orient rule (derived — apply after collecting the rows above)

When **any** loop signal trips — `harness-loop-config`, `harness-scheduled-agent`, or `harness-goal-completion-loop` (and the CI analogue `ci-scheduled-agent`) — **and** no intent-or-RETHINK instrument is present (i.e. `intent-undocumented` or `goal-drift-unmeasured` also holds), emit `loop-without-rethink` and flag the **strong-Act / stale-Orient pathology**: the project has a strong, repeating Act loop driving against an Orient step that nothing refreshes, so it keeps doing the work without ever re-asking whether the work is still the right work or the question still the right question. This is the looping form of the same intent-mechanism drift the self-audit caught on this repo (a primitive still running on a goal nobody re-checked).

- **Predicate (deterministic):** `(harness-loop-config OR harness-scheduled-agent OR harness-goal-completion-loop OR ci-scheduled-agent) AND (intent-undocumented OR goal-drift-unmeasured) AND NOT cron-disabled`.
- **Recommendation to emit:** add the **intent layer** (a written goal/charter the loop can be checked against, per `intent-alignment-audit.md`) **and** a loop-engineering RETHINK step (the goal-/question-quality re-check the loop currently lacks, per `scheduled-and-looping-primitives.md` — the loop-eng hardening). Cite both docs and the `loop-without-rethink` signal key.
- **Why both docs:** the intent doc supplies the missing *Orient* (what the loop is for); `scheduled-and-looping-primitives.md` supplies the *how* of wiring a re-check into the loop itself. Neither alone closes the gap.
- **Anti-bloat note:** this rule fetches at most `intent-alignment-audit.md` + `scheduled-and-looping-primitives.md`, and the latter is already fetched by every loop signal — so the marginal cost is one doc. ## Anti-Bloat Rule (deterministic)

The router is inclusive, but the audit degrades if too many docs dilute every recommendation. Apply this rule in order:

1. **Count signal-triggered fetches.** Exclude Always Fetch and Always Check docs from the count — they are mandatory.
2. **If count ≤ 8**, fetch all matched docs. Done.
3. **If count > 8**, drop rows in this order until count ≤ 8:
   a. Rows whose signal you are not sure you observed (any uncertainty).
   b. Rows whose signal overlaps another already-matched row (e.g., both `harness-hooks` and `harness-comprehensive` point to `harness-engineering.md` — keep the more specific one).
   c. Rows where the fetched doc is adjacent rather than central to the signal (e.g., the full `mcp-patterns.md` for a project with only one MCP server).
4. **Never drop Always Fetch or Always Check docs.**

The four intent signals (`intent-undocumented`, `goal-drift-unmeasured`, `loop-without-rethink`, `typed-memory-no-registry`) **deduplicate to one new doc** — they all fetch `analysis/intent-alignment-audit.md`, and `loop-without-rethink` additionally shares `scheduled-and-looping-primitives.md` with the loop signals that are its precondition. So even when all four trip, the intent layer adds **one** doc to the count, not four; the cluster behaves like the Unattended group above and stays within budget by construction. The Intent Interview is human-asked, not a fetch, so it never counts against the budget.

Target: 4–8 signal-triggered fetches + 3 baseline fetches = 7–11 total docs. Beyond this, the audit becomes a firehose.

## Output Requirement

In the final audit, every **Recommendation** must include:

- The analysis doc that supports it (e.g., `analysis/model-migration-anti-patterns.md`)
- The `evidence-tier` of that doc (read from the machine-readable frontmatter field, not from prose)
- The signal key that triggered the match (e.g., `model-version-4-7`)

Example:

```markdown
**Migrate implicit subagent dispatch in `.claude/agents/builder.md:14`**
- Signal: `model-version-4-7` (settings.json targets `claude-opus-4-7`)
- Source: `analysis/model-migration-anti-patterns.md` (evidence-tier: Mixed)
- Action: Replace "dispatch the work" with "Use the Explore subagent to...".
```

The `evidence-tier` is always extractable from the doc's frontmatter — look for the `evidence-tier:` line, not the prose-form tier declaration.
